# ADR-009: ContentItem Published-Only Indexing Semantics

## Status

Accepted

## Context

`ContentItem` lifecycle operations are event-driven (see ADR-008). `codex-index` consumes
those domain events and projects content state into a backend-neutral `IndexDocument` for
public search. As the lifecycle now covers seven events, this ADR establishes which events
produce indexing side effects and which do not, so future tasks do not accidentally index
draft content, working revisions, or restored-but-not-republished items.

---

## Decision

**The public index represents published content only.**

A content item appears in the index only while its status is `PUBLISHED`. Every other status
(`DRAFT`, `ARCHIVED`) means the item is not publicly searchable.

### Event Behavior Table

| Event                       | Index side effect          |
|-----------------------------|----------------------------|
| `ContentItemCreatedEvent`   | no-op                      |
| `ContentItemUpdatedEvent`   | no-op                      |
| `ContentItemPublishedEvent` | **upsert** index document  |
| `ContentItemUnpublishedEvent` | **delete** index document |
| `ContentItemArchivedEvent`  | **delete** index document  |
| `ContentItemRestoredEvent`  | no-op                      |
| `ContentItemDeletedEvent`   | **delete** index document  |

---

## Rationale per Event

### Created → no-op

A new content item starts in `DRAFT` status. Draft content is not public. No index document
should be created until the item is explicitly published.

### Updated → no-op

The current update operation modifies the working `ContentRevision` in place. The working
revision is not the published revision. Exposing draft changes in the public index before
explicit publication would violate the published-only contract.

### Published → upsert

The item transitions to `PUBLISHED` status. This is the first (and currently only) moment when
content is fit for public discovery. The subscriber loads the full item and revision values
through `ContentItemProjectionSource` to build a complete `IndexDocument`.

### Unpublished → delete

Unpublishing returns the item to `DRAFT`. The item is no longer public and must be removed from
the index.

### Archived → delete

Archiving removes the item from active circulation. The item is no longer public and must be
removed from the index.

### Restored → no-op

Restore transitions `ARCHIVED` → `DRAFT`. A restored item is not re-published automatically. It
re-enters the editorial workflow as a draft. The index document was removed when the item was
archived (see above), so no cleanup is required. If a stale document somehow persists, it will
be overwritten on the next publish.

### Deleted → delete

Hard delete permanently removes the item. Any remaining index document must be cleaned up.
The deletion subscriber must derive the `IndexDocumentId` from event fields alone — the item
is already gone from the repository when the event fires.

---

## IndexDocumentId Generation

All indexing subscribers (upsert and delete) target the same document id for a given content
item. The id is deterministic and stable:

```
content-item:{siteKey}:{contentTypeKey}:{contentItemKey}
```

Generated by `IndexDocumentIds.contentItem(siteKey, contentTypeKey, contentItemKey)` (internal
to `codex-index`). Using the same format for upsert and delete guarantees that removal
subscribers always target the exact document written by the publish subscriber.

---

## Projection Reload Behavior

| Subscriber                                  | Reloads from repository? | Reason                                                           |
|---------------------------------------------|--------------------------|------------------------------------------------------------------|
| `ContentItemPublishedIndexingSubscriber`    | **yes** (via `ContentItemProjectionSource`) | needs full field values and revision data to build `IndexDocument` |
| `ContentItemUnpublishedIndexingSubscriber`  | no                       | only needs id; computes it from event fields                     |
| `ContentItemArchivedIndexingSubscriber`     | no                       | only needs id; computes it from event fields                     |
| `ContentItemDeletedIndexingSubscriber`      | no — **must not**        | hard delete: item no longer exists when the event fires          |
| `ContentItemRestoredIndexingSubscriber`     | n/a — not implemented    | no-op for now                                                    |

## Known Limitations and Future Follow-ups

### No draft index

Draft and working-revision content is not searchable. An editorial/preview index would require
a separate index region and separate event routing. Not in scope for the current published-only
model.

### No preview or editorial index

Editors cannot currently search unpublished content through `codex-index`. A future editorial
index may consume `ContentItemCreatedEvent` and `ContentItemUpdatedEvent` and route to a
separate non-public index region.

### No reindex-on-update for already-published items

When `ContentItemUpdatedEvent` fires, the public index is stale if the item was previously
published. The update operation currently modifies the working revision only; once edit/revision
fork semantics are defined (separate working and published revisions), a re-publish event will
drive the index update. Updating the index from `ContentItemUpdatedEvent` directly would
expose draft changes, which is intentionally disallowed.

### No bulk rebuild

There is no `reindexAll` or `rebuildIndex` operation. If the index becomes inconsistent (e.g.,
after a storage migration or subscriber outage), a bulk rebuild task must be added separately.

### No search permission or visibility model

The public index has no per-site, per-tenant, or per-role visibility filter. All indexed
documents are treated as equally public. A future permission layer may need to partition the
index by site or attach visibility metadata to each document.

### No multi-language or variant index strategy

Content variants and language editions are not yet modeled. The index document format will need
to evolve when multi-language support is added.
