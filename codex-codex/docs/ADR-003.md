# ADR-003: Event Dispatching and Transaction Context

## Status

Draft

## Context

Codex services need to publish events without depending on a specific runtime, framework, message broker, or transaction manager.

State-changing service operations, such as creating a site or changing a site status, may eventually run inside a transaction boundary. Events emitted by those operations should not be delivered too early if the enclosing operation later fails or rolls back.

At the same time, Codex should support different delivery mechanisms:

- local in-memory dispatch
- synchronous subscribers
- asynchronous subscribers
- pub/sub systems
- Kafka or other brokers
- cache invalidation
- audit recording
- projections
- workflow continuation

The event system must remain framework-agnostic. It should not depend directly on Spring, `@Transactional`, Kafka, Redis, or any specific infrastructure.

## Decision

Codex introduces a lightweight transaction-aware event dispatching model based on three concepts:

```text
CodexEvent
CodexEventDispatcher
TransactionContext
```

Service-level code may dispatch events through CodexEventDispatcher without knowing whether a transaction context is currently active.

A transaction-aware dispatcher, currently represented by DeferredEventDispatcher, decides whether to deliver the event immediately or defer it until the active transaction context commits.

Transaction Context

Codex uses TransactionContext as a lightweight execution-scope context.

The context is based on Java ScopedValue, not ThreadLocal.

This allows Codex to carry transaction-scoped state without depending on Spring or another transaction framework.

A transaction context can:

- report whether a context is active
- expose the current context
- register transaction callbacks
- store per-transaction attachments
- notify callbacks on commit
- notify callbacks on rollback

The core contract is:
```
run action inside transaction context
  if action succeeds:
    notify commit callbacks
  if action fails:
    notify rollback callbacks
    rethrow original exception
```

Commit and rollback callbacks are intentionally generic. They are not limited to events.

Potential future uses include:

- deferred event buffers
- rollback cleanup actions
- temporary resource cleanup
- lock release
- audit buffering
- unit-of-work style resources
- Event Dispatching

Codex events are dispatched through:
```
public interface CodexEventDispatcher {
    void dispatch(CodexEvent event);
}
```
This interface does not define whether delivery is local, remote, synchronous, asynchronous, immediate, deferred, broker-based, or subscriber-based.

Those concerns belong to implementations and decorators.

# Deferred Event Dispatching

DeferredEventDispatcher wraps another CodexEventDispatcher.

Its responsibility is to decide when an event should be delivered.

The behavior is:

```
Outside a TransactionContext:
  dispatch immediately

Inside a TransactionContext:
  buffer event in the current transaction context
  dispatch buffered events only on commit
  discard buffered events on rollback
```

This allows service code to remain simple:

```
eventDispatcher.dispatch(event);
```

while the dispatcher decides whether the event is safe to deliver immediately or must wait for commit.

# Dispatch Mode

Codex supports dispatch modes:

```
SYNC
ASYNC
```

The dispatch mode determines how the event is executed once it is ready to be delivered.

Dispatch mode resolution follows this order:
```
1. EventEnvelope explicit mode
2. AsyncCodexEvent marker
3. default SYNC
```

This allows callers or event types to request asynchronous delivery without hardcoding delivery behavior into every dispatcher.

# EventEnvelope

EventEnvelope can wrap a CodexEvent and provide explicit dispatch metadata.

The envelope takes precedence over marker interfaces.

This allows a caller to override delivery behavior for a specific dispatch operation without changing the event type itself.

# Async Events

Events implementing AsyncCodexEvent are treated as asynchronous by default.

Asynchronous execution is delegated to CodexExecutor.

This keeps event dispatching independent from a specific executor implementation.

A runtime may provide an executor backed by:

- virtual threads
- platform threads
- bounded queues
- managed executors
- test executors
- custom backpressure strategies

# Virtual Threads

Codex may use virtual threads for asynchronous event execution.

Virtual threads are appropriate for many event-handling workloads, but they do not remove the need for backpressure.

Future executor implementations may need to consider:

- queue limits
- rejection policy
- subscriber isolation
- timeout handling
- error handling
- structured concurrency
- context propagation

The event itself should contain the information needed by asynchronous subscribers. Async subscribers should not assume that TransactionContext or other scoped runtime context remains active.

# Rollback Semantics

When the transaction action fails, rollback callbacks are invoked.

Events buffered during the transaction are discarded on rollback.

Rollback callbacks should not attempt to magically undo arbitrary external side effects. They are mainly useful for local cleanup and compensating infrastructure actions.

Future use cases may include:

- deleting temporary files
- releasing local resources
- clearing staged buffers
- marking deferred work as cancelled

For distributed or external side effects, Codex should prefer explicit compensating actions over pretending that rollback can undo everything.

# Commit Semantics

Events buffered inside a transaction are delivered after the transaction action completes successfully.

A commit callback failure should not be treated as a failure of the original transaction action.

The exact policy for commit callback failures is still open and may evolve.

Possible future policies include:

```
fail fast
collect and suppress exceptions
log and continue
isolate each subscriber
retry failed delivery
send failed events to a dead-letter channel
```

For now, the important semantic rule is:

```
rollback callbacks are only for failed transaction actions
commit callback failures must not trigger rollback semantics
```

# Subscriber Model

Codex may later support subscribers such as:
```
public interface CodexEventSubscriber<E extends CodexEvent> {
    void onEvent(E event);
}
```

A future Java module-based subscription model may support annotation-based event subscription.

For example, a method could declare interest in a specific event type and be registered by the runtime.

This should remain independent from Spring’s event model.

Spring may later provide an adapter, but Codex core should not require Spring events.

# Events vs Hooks

Events and hooks are different concepts.

```
Events
  facts that already happened

Hooks / policies / operation pipeline
  extension points around something happening
```

Events should generally not decide whether an operation is allowed to proceed.

Future before/after/around hooks may support policy validation, extension logic, scripting, or custom module behavior.

Events are primarily for coordination after a fact has occurred.

# Relationship to Audit, Cache, and Projections

Events may become the coordination point for multiple mechanisms:

- audit recording
- cache invalidation
- read-model updates
- projections
- workflow continuation
- notifications
- external broker publication

For example:

```
SiteCreatedEvent
  -> audit record
  -> cache invalidation
  -> projection update
  -> external notification
```
or:
```
SiteStatusChangedEvent
  -> audit status transition
  -> invalidate site cache
  -> update lookup projections
```

This does not mean every event must always go to every mechanism. The event pipeline should allow independent subscribers.

# Consequences
## Positive
Codex services can dispatch events without knowing about transaction infrastructure.
Events are not delivered before a successful transaction commit.
Rollback discards pending events.
Sync and async delivery are modeled explicitly.
Event delivery remains framework-agnostic.
Spring can be used later without infecting Codex core.
Kafka, pub/sub, local listeners, audit, cache invalidation, and projections can be introduced as adapters.
ScopedValue provides a modern execution-scope context without ThreadLocal.

# Negative / Tradeoffs
TransactionContext introduces infrastructure that must be carefully tested.
Async delivery requires clear failure-handling rules.
ScopedValue requires care when crossing thread boundaries.
Deferred event delivery adds complexity compared to immediate dispatch.
Callback ordering and failure handling must be documented and evolved carefully.
The current model is not a replacement for a real database transaction manager.

# Current Implementation Direction
The current implementation includes:

```
TransactionContext
TransactionCallback
CodexEvent
CodexEventDispatcher
DeferredEventDispatcher
DispatchMode
AsyncCodexEvent
EventEnvelope
CodexExecutor
```

The current model is sufficient for:

buffering events during a Codex transaction context
dispatching after commit
discarding on rollback
supporting sync and async event delivery
testing event behavior without Spring

# Guiding Principle

Codex events should describe facts that happened.

Codex dispatchers decide when and how those facts are delivered.

Codex runtimes decide how delivery integrates with transactions, threads, brokers, audit, cache, observability, and external systems.
