

# ADR-001 — Core Ownership of `codex-codex`

## Status

Accepted

## Context

As Codex begins to move from lore and module skeleton into actual domain modeling, we need to define what belongs in `codex-codex`, what does not belong there, and how the central content model should be shaped.

This decision is important because several other modules depend conceptually on the answer:

- `Chronicon` depends on a clear notion of revision and history
- `Custos` depends on a clear boundary between content and identity
- `Iter` depends on a clear lifecycle target
- `Archivum` depends on a clean domain model rather than a persistence-driven design

The goal is to avoid a “god entity” design like the contentlet model seen in older systems, where identity, current state, storage metadata, permissions, revisioning, and content payload become entangled in one mutable structure.

## Decision

`codex-codex` is the semantic core of Codex.

Its responsibility is to answer the question:

**What is content in Codex?**

From that perspective, `codex-codex` owns the foundational content language of the system.

### It owns

The following concepts belong to `codex-codex`:

- `Site`
- `ContentType`
- `ContentTypeVersion`
- `ContentItem`
- `ContentRevision`
- editorial states
- content identities
- content-oriented value objects

### It does not own

The following do not belong in `codex-codex`:

- persistence strategy
- JPA, JDBC, JSONB, or storage mappings
- REST controllers
- search integration
- workflow engine implementation
- authorization rules
- scripting runtime
- AI / agentic behavior
- infrastructure concerns in general

Those concerns belong to other modules.

## Detailed Decisions

### 1. `ContentType` and `ContentTypeVersion` are separate concepts

We separate:

- `ContentType` as the stable logical identity of a type
- `ContentTypeVersion` as a concrete versioned schema definition

Rationale:

A single type concept should not be forced to represent all of the following at once:

- stable identity
- version number
- field definitions
- schema history
- current active definition

This separation keeps the model clearer and allows `ContentItem` to reference the type definition under which it was created or validated.

### 2. `ContentType` belongs to a `Site`

`ContentType` is scoped to a `Site`.

That site may be:

- `_system`
- or a concrete tenant/site

This supports both globally shared type definitions and site-specific type definitions without introducing special-case logic.

### 3. `Site` is not modeled as content

`Site` is a first-level entity of its own.

Codex explicitly rejects the design where `Site` is modeled as a specialized content item just to reuse versioning, permissions, publishing, tree, or similar capabilities.

Rationale:

That approach can appear elegant at first, but in practice it tends to produce special-case branching, leaky abstractions, and method-level `if` statements for operations such as archive, delete, or lifecycle handling.

Instead, Codex prefers:

- composition
- shared capabilities
- conceptual interfaces / traits
- reusable services

rather than forcing unrelated concepts into the same inheritance chain.

### 4. `ContentItem` and `ContentRevision` are separate concepts

We separate:

- `ContentItem` as the stable logical identity of a content entry
- `ContentRevision` as the concrete snapshot that holds the actual versioned information

This is a central design decision.

#### `ContentItem` represents:

- the identity of the content entry
- its site ownership
- its type reference
- its variant identity
- its stable metadata
- pointers to live or working revisions, if needed by the model

#### `ContentRevision` represents:

- revision number
- title
- tags
- payload / field values
- creation metadata
- editorial state
- change history metadata if needed

This means the revision is the structure that actually carries the evolving information, while the content item is the stable pivot around which those revisions exist.

### 5. Title and tags belong to the revision

The initial modeling decision is that:

- `title` lives on `ContentRevision`
- `tags` live on `ContentRevision`

Rationale:

These are part of the content snapshot and may change over time as the content evolves.

### 6. Identities must be explicit and semantically meaningful

Codex prefers explicit identity types over anonymous raw strings or UUIDs scattered throughout the model.

Candidate identity and value-object types include:

- `SiteId`
- `ContentTypeId`
- `ContentTypeVersionId`
- `ContentItemId`
- `ContentRevisionId`
- `VariantGroupId`
- `Locale`
- `Slug`
- `EditorialState`
- `ActorId` (possibly later extracted if it becomes more broadly shared)

The purpose of these types is to make the domain more expressive and less error-prone.

### 7. Identity should not be casually coupled to mutable names

Codex rejects using a display name as the true identity of a content type.

A type may need:

- a stable identity
- a readable technical key
- a human-visible display name
- optional aliases or migration support later

Relying on a mutable human-facing name as the true identifier creates painful rename scenarios and unnecessary coupling.

### 8. Deterministic identity is preferred as a direction, but not yet frozen in full detail

Codex prefers identity models that are reproducible and grounded in domain meaning, rather than blindly depending on random UUIDs everywhere.

However, the exact strategy is not fully frozen yet.

The current architectural preference is:

- avoid UUIDs as the only conceptual notion of identity
- move toward deterministic or structured identity where appropriate
- keep the design open while the model matures

A practical concern was identified:

- if an identity depends on mutable elements, rename and reference stability become difficult

This means identity design must be deliberate. The system should distinguish between:

- internal identity
- readable keys
- hierarchical location / path
- display names

### 9. Treeability is a useful concept, but not a reason to collapse everything into content

Codex acknowledges the usefulness of a hierarchical model.

The content world may eventually be navigable through tree-like structures such as:

- `site/folder/content-item`
- `system/folder/content-item`
- potentially version-aware paths for internal navigation

This is compatible with the idea that content is treeable, but it does **not** imply that all first-level entities must become content items.

Tree-like navigation is a capability, not a sufficient reason to merge distinct domain concepts.

### 10. `ActorId` may start in `codex-codex`

The content model needs lightweight references to actors such as:

- created by
- modified by
- published by

For now, `ActorId` may live in `codex-codex` if that keeps the model simple during early iterations.

If it later proves to be a genuinely shared foundational abstraction across multiple modules, it may be extracted or relocated.

## Consequences

### Positive

- the core content model remains clear and focused
- Codex avoids a contentlet-style god entity
- type versioning becomes explicit
- revision-aware behavior becomes natural
- the identity boundary with `Custos` stays cleaner
- `Site` remains a true first-level concept
- later modules can depend on a stronger semantic core

### Tradeoffs

- the model introduces more types up front
- some decisions about identity strategy remain intentionally open
- treeability and shared capabilities will need careful design later
- some cross-cutting abstractions may migrate once the system matures

## Notes for Future ADRs

This ADR intentionally leaves some topics open for later refinement:

- exact identity generation strategy
- exact shape of `ContentTypeKey`, `Slug`, and aliases
- tree model and hierarchical addressing
- capability interfaces such as `Permissionable`, `Treeable`, `Versionable`, or equivalent traits
- publication pointers and live/working revision semantics
- whether `ActorId` stays here or moves into a more foundational module
