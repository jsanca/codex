# SiteService Internal Implementation

Default internal implementation of `SiteService`.

This class contains the first application-level semantics for site operations: creation, lookup, status transitions, and alias resolution. It intentionally remains framework-agnostic and does not depend on Spring, HTTP, persistence annotations, or authentication-specific concepts.

## Future Service-Level Concerns

Codex services are expected to evolve with the following cross-cutting capabilities. These concerns should be introduced through explicit ports, decorators, adapters, or runtime composition rather than by coupling this service directly to a specific framework.

* **Transactions:** Service operations that read, validate, mutate, and save state should eventually run inside a transaction boundary.
* **Events:** State-changing operations should eventually publish application/domain events. The exact event model, event timing, and commit semantics still need to be designed.
* **Contention control:** State transitions should eventually be protected by a lock or concurrency-control mechanism. A local runtime may use a JVM lock, while clustered runtimes may use ShedLock, database locks, Redis, or another distributed locking strategy.
* **Auditability:** Operations should be auditable. Logging is the first lightweight mechanism, but the audit model, actor representation, correlation identifiers, and sensitive-data rules still need to be designed.
* **Caching:** Read-heavy services may later use caches, but cache boundaries, invalidation, and consistency rules must be decided per use case.
* **Indexes:** Not every lookup must always be modeled as a direct database hit. Some services may use indexes or projections. For site lookup, this may be less important initially, but the design should not close the door to indexed resolution later.
# ADR-002: SiteService Internal Implementation and Future Service Pipeline

## Status

Draft

## Context

`CodexSiteService` is the default internal implementation of `SiteService`.

This service contains the first application-level semantics for site operations: creation, lookup, status transitions, unarchiving, and alias resolution. It intentionally remains framework-agnostic and does not depend on Spring, HTTP, persistence annotations, authentication-specific concepts, or a specific transaction manager.

`SiteService` is not intended to be a mirror of `SiteRepository`. The repository owns storage concerns. The service owns application semantics: state transitions, actor-aware operations, precondition validation, and future integration points such as events, audit, locks, cache invalidation, and extension hooks.

## Decision

Codex services should remain small, explicit, and framework-agnostic. Cross-cutting capabilities should be introduced through explicit ports, decorators, adapters, or runtime composition rather than by coupling core services to a specific framework.

A future service pipeline may be composed through wrappers/decorators such as:

```text
TransactionalSiteService
  -> LockingSiteService
    -> AuditingSiteService
      -> EventDispatchingSiteService
        -> CodexSiteService
```

The exact order still needs to be validated per use case, but the important decision is that these concerns should be composable rather than hardcoded into `CodexSiteService`.

## Future Service-Level Concerns

### 1. Transactions

Transactions are a runtime/container concern.

Codex core should not depend on Spring `@Transactional`, JTA, JDBC transaction APIs, or any specific transaction mechanism. A service operation should be designed so it can be wrapped by a transaction boundary, but the core service should not own the transaction implementation.

A Spring runtime may apply transactions using Spring configuration or decorators. Another runtime may provide a different transaction wrapper.

### 2. Events

State-changing operations should eventually dispatch Codex events through an implementation-agnostic event pipeline.

Events should not assume whether they are delivered through an in-memory broadcaster, local subscribers, a pub/sub system, Kafka, Redis, a database outbox, or another mechanism.

A possible future port is:

```java
public interface CodexEventDispatcher {
    void dispatch(CodexEvent event);
}
```

A future subscription model may support explicit subscribers such as:

```java
public interface CodexEventSubscriber<E extends CodexEvent> {
    void onEvent(E event);
}
```

Codex may later support annotation-based subscription for Java modules, allowing methods to subscribe to specific event types.

Events are important because they can become the coordination point for multiple mechanisms:

- local broadcasting
- audit recording
- cache invalidation
- projections or lookup indexes
- external pub/sub publication
- workflow continuation
- notifications

Events represent facts that already happened. They should be distinguished from hooks or policies that may run before an operation and possibly block it.

### 3. Contention Control / Locks

State transitions should eventually be protected by a contention-control mechanism.

A local runtime may use JVM-level locking. A clustered runtime may use ShedLock, database locks, Redis locks, advisory locks, or another distributed locking strategy.

A possible future port is:

```java
public interface CodexLockRunner {
    <T> T runWithLock(String lockName, Supplier<T> operation);
}
```

The preferred integration style is a decorator/wrapper rather than hardcoding locking inside the core service:

```text
LockingSiteService
  -> CodexSiteService
```

This keeps the business implementation focused while allowing different runtimes to provide different locking strategies.

### 4. Auditability

Auditability is a Codex concern, not merely an observability concern.

Audit records describe business-relevant facts such as who performed an operation, what resource was affected, what changed, and when it happened. Audit data may eventually have its own datasource, schema, retention rules, and query model.

For example, a site status transition audit record may include:

```text
actor = jsanca
resource = site:main-site
action = site.suspended
previousStatus = STARTED
newStatus = SUSPENDED
occurredAt = ...
correlationId = ...
```

Logging can be a lightweight first step, but logs should not be the only audit model.

### 5. Observability

Observability is primarily an operations/devops concern.

Metrics, traces, structured logs, and dashboards may be implemented through tools such as Prometheus, Grafana, OpenTelemetry, log aggregation, or platform-specific mechanisms.

Observability should remain implementation-agnostic from Codex core. It may be introduced through wrappers, event subscribers, instrumentation decorators, or runtime adapters.

### 6. Caching

Caching should be implementation-agnostic and introduced through explicit cache decorators or repository/service wrappers.

Codex may later support a layered cache pipeline, for example:

```text
L1 cache: Caffeine
L2 cache: Redis
Source of truth: repository/database
```

Cache invalidation should preferably be coordinated through events. For example, a `SiteUpdatedEvent` or `SiteStatusChangedEvent` could invalidate site lookup cache entries, alias mappings, routing cache, or projections.

Caching should be decided per use case. It may be more important for content rendering, content types, field schemas, and permission checks than for the first version of `SiteService`.

### 7. Indexes and Projections

Not every lookup must always be modeled as a direct database hit.

Some services may use indexes, lookup tables, projections, or read models. For `SiteService`, this is probably not a major concern at the beginning, except perhaps for alias resolution.

The current design should not force indexes into `CodexSiteService`, but it should not close the door to indexed resolution later.

### 8. Extension Hooks / Operation Pipeline

Codex should eventually support service-level extension points.

The term `interceptor` may be too specific or may carry framework assumptions. A more general concept is an operation pipeline or service hook system.

Possible extension points include:

- before operation hooks
- after operation hooks
- around operation wrappers
- validation/policy hooks
- Java module-based extensions
- later scripting support, such as JavaScript-based custom logic

A hook system should be explicit, ordered, observable, and safe. It should define clearly whether a hook can only observe, can block an operation, can mutate inputs, can mutate outputs, or can run custom code.

A possible evolution path is:

```text
1. Static Java module extensions
2. Explicit service hooks or decorators
3. Event subscribers
4. Optional scripting support for custom behavior
```

Events and hooks should remain conceptually separate:

```text
Events: facts that happened.
Hooks/policies: extension points around something happening.
```

## Consequences

- `CodexSiteService` remains focused on application semantics.
- Framework-specific concerns stay outside the core service.
- Spring can be used later without making Codex depend on Spring concepts.
- Other runtimes can provide their own transaction, lock, cache, event, audit, or observability mechanisms.
- The system can evolve toward a composable service pipeline without forcing that complexity into the first implementation.
