# ADR-004: System Site and Lifecycle Participation

## Status

Accepted

## Context

Codex needs a built-in platform site to act as the scope for global resources — content types,
field definitions, global settings, and other artefacts that do not belong to a specific
user-managed tenant site.

At the same time, services that manage site lifecycle (start, suspend, archive, unarchive)
must not allow those operations on platform-managed infrastructure.

The naive approach — checking `site.key().equals(SiteKey.SYSTEM)` inside every service —
is fragile: it hardcodes a specific key and cannot generalize to future resource types such
as read-only resources, externally managed resources, or virtual resources.

## Decision

### Lifecycle Participation

Introduce `LifecycleParticipation` as a first-class enum in `codex-fundamentum`:

```
MANAGED          — participates in the normal Codex lifecycle
READ_ONLY        — visible and addressable, not mutable through normal services
SYSTEM_MANAGED   — managed by the Codex runtime; lifecycle operations rejected
EXTERNAL         — lifecycle is owned outside Codex
```

`Site` carries a `lifecycleParticipation` field that defaults to `MANAGED` for regular sites.

### Validation Pipeline

`CodexSiteService` validates lifecycle participation before any status transition.
The validation order is:

1. Reject if `lifecycleParticipation != MANAGED` → `InvalidSiteLifecycleOperationException`
2. Return existing site if status already equals target (idempotent)
3. Reject invalid status transitions → `InvalidSiteStatusTransitionException`

Step 1 applies even for idempotent operations: calling `start` on a `SYSTEM_MANAGED` site
throws, not silently returns. This ensures that access to non-managed sites is always auditable
and explicit.

The check is based on `LifecycleParticipation`, not on `SiteKey.SYSTEM`. A site with any key
whose participation is not `MANAGED` is rejected identically.

### System Site

`SiteKey.SYSTEM` is a strongly typed constant (`"system"`) declared in `SiteKey`.

`SiteId.system()` returns a stable, deterministic identifier (`"system"`) for the platform site.

`Site.system()` is a factory method — not a mutable static field — that returns a new instance
representing the built-in platform site:

```
key                  = SiteKey.SYSTEM
id                   = SiteId.system()
displayName          = "System"
status               = STARTED
lifecycleParticipation = SYSTEM_MANAGED
```

The system site does not require persistence. Runtime preloading, repository decoration,
or a resource resolver may be introduced later.

### Global Resources

Global resources (content types not scoped to a tenant, global field definitions, platform
settings) will use `SiteKey.SYSTEM` as their site scope. This keeps the data model consistent:
every content-scoped resource has an owning site key, and global resources simply belong to
the system site.

## Consequences

### Positive

- Services validate lifecycle semantically, not by key comparison.
- New resource kinds (read-only, external, virtual) can be supported by extending `LifecycleParticipation`.
- `Site.system()` is immutable and stateless; it can be constructed anywhere without infrastructure.
- Global resource scoping is consistent: all resources have a site scope.
- Tests can prove behaviour using arbitrary keys with `SYSTEM_MANAGED` participation.

### Negative / Tradeoffs

- `Site` gains a new record field; existing builders and `copyOf` callers are unaffected because
  the canonical constructor defaults `lifecycleParticipation` to `MANAGED` when null.
- The system site is not automatically persisted; a lookup by key will return empty until a
  runtime preloads it or a repository decorator intercepts the lookup.

## Future Work

- Runtime preloading or repository decorator for the system site.
- `TreeableResource` and `CodexResourcePath` for hierarchical resource navigation.
- Resource protocols (e.g. `codex://system/content-types/blog-post`).
- Extend `LifecycleParticipation` validation to `ContentTypeService` and other services.
- Permission service integration: participation may inform read/write access rules.
