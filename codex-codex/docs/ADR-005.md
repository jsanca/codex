# ADR-005: ContentType Event Publishing Pipeline

## Status

Accepted

## Context

ContentType operations (create, activate, archive) must produce observable domain events following
the same decorator-based pipeline already established for Site events (see ADR-003).

## Decision

ContentType event publishing uses the same pattern as Site event publishing:

- `CodexContentTypeService` owns business semantics only. It has no knowledge of events.
- `EventPublishingContentTypeService` wraps the core service and dispatches events after successful mutations.
- Both decorators share the single `DeferredEventDispatcher` instance wired in `CodexRuntime`.

Pipeline:

```
ContentTypeService call
  -> EventPublishingContentTypeService
     -> CodexContentTypeService
        -> MemoryContentTypeRepository

Event dispatch
  EventPublishingContentTypeService
    -> DeferredEventDispatcher
       -> shared event delegate / recorder / subscribers
```

Events dispatched:

| Operation | Event                      | Condition                      |
|-----------|----------------------------|--------------------------------|
| create    | `ContentTypeCreatedEvent`  | always after successful create |
| activate  | `ContentTypeActivatedEvent`| only when status actually changes |
| archive   | `ContentTypeArchivedEvent` | only when status actually changes |

## Consequences

- Idempotent operations (e.g. activating an already-ACTIVE content type) produce no event.
- Invalid operations (e.g. activating an ARCHIVED content type) throw before any event is dispatched.
- Rollbacks discard all buffered events via `DeferredEventDispatcher` and `TransactionContext`.
- Outside a transaction, events are dispatched immediately.
- Event subscribers, audit, cache invalidation, workflow continuation, and external brokers (Kafka, outbox)
  are future work and must not be implemented here.
